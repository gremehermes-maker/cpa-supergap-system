# CPA 시스템 한계점 및 해결방안 합의서 (Review & Solutions)

> **💡 [문서 개요]**
> 본 문서는 `CPA_인수인계_통합_마스터플랜_최종본.md`에 내포된 구조적, 운영적, 인지적 한계점들을 5대 대분류로 점검하고, 사용자(수험생)와의 심층 논의를 통해 확정된 **최종 해결책(Solution)** 및 **수용(Pass) 정책**을 아주 상세하게 기록한 부속 참조 문서입니다. 향후 파이썬 스크립트 수정 및 에이전트 구축 시 이 합의서를 기준으로 개발을 진행합니다.

## 1. 외부 생태계 종속성 및 인프라 가용성 리스크

* **1.1 MarginNote URL 스킴 종속성 리스크**
  * **상세 내용:** 모든 오답 연결망과 메타인지 통계가 MarginNote의 고유 URL(`marginnote4app://`)에 기반함. 애플의 샌드박스 정책 변경이나 MarginNote 앱 업데이트 시 URL 스킴 작동이 중단될 위험 존재.
  * **최종 합의:** **수용 (패스)**
  * **사유:** 애플 생태계의 폐쇄성과 앱 종속성은 어쩔 수 없는 한계로 인정하며, 현재 환경에서 가장 효율적인 방식이므로 리스크를 안고 감.

* **1.2 비공식 DB 파싱 취약성 (스키마 변경 대비)**
  * **상세 내용:** MarginNote 메이저 업데이트 시 내부 `.marginnotes` SQLite 테이블 스키마(예: `ZBOOKNOTE`, `ZBOOKCOMMENT`)가 약간만 변경되어도 파이썬 추출 스크립트에 치명적 에러 발생 위험.
  * **최종 합의:** **승인 및 방어 로직 도입**
  * **해결책 상세 (Sanity Check):** 파이썬 스크립트 실행 시 추출 작업 전 가장 먼저 `PRAGMA table_info(ZBOOKNOTE)` 명령을 실행. 우리가 사용하는 핵심 컬럼(`ZNOTEID`, `ZCOMMENTTEXT` 등)의 존재 여부를 검사하는 무결성 검증 단계를 거침. 스키마 변경이 감지되면 즉각 동기화를 중단하고 Slack/Telegram 봇으로 **"🚨 [긴급] MarginNote DB 스키마 변경 감지! 데이터 꼬임을 막기 위해 동기화를 강제 중단합니다"** 라는 강력한 알람 송출.

* **1.3 로컬 PC(Mac) 서버 의존성**
  * **상세 내용:** 폰에서 내린 명령을 처리하기 위해 Mac이 상시 서버처럼 작동해야 함. 외출 시 절전모드 등으로 인한 가용성 공백 우려.
  * **최종 합의:** **문제 없음 (클리어)**
  * **사유:** 사용자는 헤르메스 에이전트 가동을 위해 Mac을 상시 켜두는 환경을 유지하므로, 시스템 휴지기(Downtime) 이슈는 발생하지 않음.

* **1.4 AnkiConnect 포트 침묵 실패(Silent Fail)**
  * **상세 내용:** Mac에서 자동화 스크립트는 도는데 정작 Anki 프로그램이 꺼져있거나 크래시 나서 포트(8765)가 닫힌 경우, 에러 없이 데이터 동기화가 누락되는(Silent Fail) 치명적 문제.
  * **최종 합의:** **승인 및 자동 복구/알람 도입**
  * **해결책 상세 (Health Check):** 파이썬 동기화 스크립트 구동 시 `http://localhost:8765` 로 핑(Ping) 테스트 수행. 무응답 시 Mac OS 내부 명령어(`open -a Anki`)를 호출해 백그라운드에서 Anki를 강제 실행. 만약 재실행 후에도 핑 실패 시, Slack 봇을 통해 **"🚨 Anki 앱이 꺼져 있어 동기화가 중단되었습니다."** 라고 즉시 알람을 발생시켜 침묵 실패를 완벽히 차단.

## 2. 데이터 정합성 및 백업 시스템 리스크

* **2.1 단방향 동기화 및 모바일 수정 데이터 증발**
  * **상세 내용:** 스마트폰(AnkiDroid)에서 데이터를 수정한 후 PC와 동기화 시, 충돌 병합 없이 PC 원본 데이터로 덮어씌워져 모바일 수정 내역이 날아감.
  * **최종 합의:** **수용 (클리어)**
  * **사유:** "안드로이드(폰)에서는 오직 읽고 복습만 하며 절대 카드 내용을 수정하지 않는다"는 강력한 사용자 행동 원칙으로 해결.

* **2.2 분산 DB 간 시점 불일치(Time-Sync Mismatch)**
  * **상세 내용:** MarginNote DB, Obsidian 폴더, Anki DB 3개가 각각 구동됨. 시스템 장애 시 5TB 구글 드라이브에서 백업본을 롤백할 때, 특정 앱만 예전 백업본으로 덮어씌우면 DB 간 참조가 깨져 고아 데이터(Orphan Data)가 대량 발생.
  * **최종 합의:** **승인 및 원자적 백업(Atomic Backup)**
  * **해결책 상세:** 백업을 수행할 때 스크립트가 1) 마진노트 `.marginnotes` 원본, 2) Obsidian Vault 마크다운 폴더, 3) Anki `collection.anki2` 파일을 **반드시 한 날 한 시에 하나의 단일 `.zip` 파일**로 압축하여 저장함. 문제 발생 시 사용자는 해당 통합 zip 파일 압축을 풀어 3곳에 동시에 덮어씌움으로써 시점 불일치를 원천 차단.

* **2.3 개정판 출시에 따른 연결망 붕괴 (Stale Data)**
  * **상세 내용:** 오답 연결 시 목차 번호(예: `1.1.1 자산손상`)에 과도하게 의존할 경우, 내년 교재 개정으로 번호가 밀리면 기존에 쌓인 Obsidian 약점 랭킹 통계와 연결망이 모두 붕괴됨.
  * **최종 합의:** **승인 및 버전 관리 파일 분리**
  * **해결책 상세:** 시스템 내부나 에이전트 프롬프트에 `1.1.1`을 하드코딩하지 않음. 대신 `TOC_Mapping.json` 이라는 별도의 맵핑 파일을 생성. 
  ```json
  {
    "1.1.1": "자산손상",
    "1.1.2": "법인세"
  }
  ```
  에이전트는 사용자 명령 시 무조건 이 json 파일을 조회하여 목차를 해석함. 교재 개정 시 json 파일 내 번호표만 한 번 싹 고쳐주면, 기존 연결망을 훼손하지 않고 전체 시스템이 새 교재 버전에 즉각 동기화됨.

* **2.4 수동 조작 시 UUID 파괴 및 카드 중복 증식**
  * **상세 내용:** 파이썬 스크립트가 카드의 중복 방지를 위해 심어둔 마진노트 고유 ID(`ZNOTEID`)를, 사용자가 Anki 내에서 수동으로 '카드 복사' 또는 '잘라내기(Split)' 기능으로 건드릴 경우 UUID가 파괴되거나 복제되어 카드가 무한 증식하는 문제.
  * **최종 합의:** **수용 (클리어)**
  * **사유:** 2.1항과 동일한 맥락에서 "Anki Desktop 내에서는 절대 시스템 생성 카드를 수동으로 쪼개거나 내용을 복사 생성하지 않는다"는 대원칙 하에 자연스럽게 방어. 분할이 필요하면 오직 MarginNote 원본에서 분할함.

## 3. AI 에이전트 및 RAG 파이프라인 리스크

* **3.1 전문 용어 STT 변환의 품질 저하 (Data Poisoning)**
  * **상세 내용:** 회계/세법 강의의 복잡한 한자어 및 강사의 억양으로 인해 STT가 오탈자를 만들고, 이 오염된 텍스트가 Tutor 에이전트의 지식 베이스(ChromaDB)에 적재되어 심각한 환각(Hallucination)을 유발할 위험.
  * **최종 합의:** **승인 및 Ground-Truth 기반 오토 코렉트 파이프라인 구축**
  * **해결책 상세:** 이미 보유 중인 가장 완벽한 텍스트 원본(교재 PDF 텍스트 및 마진노트 데이터)에서 **'사용자 전용 용어 사전(Custom Dictionary)'**을 자동 추출함. STT 텍스트가 RAG DB에 적재되기 전 중간 필터 역할을 하는 소형 LLM(또는 파이썬 스크립트)이 이 사전을 기반으로 오탈자("자본유지 젓갈법" -> "자본유지 접근법")를 강제 치환 및 정제한 후 최종 저장함. 이를 통해 RAG의 순도를 최고 수준으로 유지.

* **3.2 요약 에이전트로 인한 미세 컨텍스트 소실**
  * **상세 내용:** Memory Editor 에이전트가 1만 토큰의 긴 슬랙 대화를 단축시키기 위해 핵심만 뽑아 한 줄로 요약하는 과정에서, 세법의 미세한 예외 조건이나 사용자 오답의 섬세한 뉘앙스가 영구 소실됨.
  * **최종 합의:** **승인 및 구조화된 프롬프트 양식(JSON/YAML) 강제**
  * **해결책 상세:** Memory Editor 에이전트에게 절대 자유도 높은 산문형 한 줄 요약을 허용하지 않음. 하드디스크(`LLM_Wiki/User_State.md`)에 기록할 때 아래와 같은 구조화된 포맷 출력을 강제함.
    * `[약점 개념]`: 단기매매금융자산
    * `[결정적 조건/가정]`: 시장성 상실 시
    * `[사용자의 실수 원인]`: 평가손익을 당기손익이 아닌 OCI로 잘못 인식함
  이처럼 틀을 강제하여 미세한 컨텍스트 소실을 방지하고 기계 가독성(Machine Readability)도 확보.

* **3.3 Checker/Tutor 교차 검증의 무한 루프**
  * **최종 합의:** **보류 (패스)**
  * **사유:** Claude와 Qwen이 서로 합의하지 못하고 교착 상태(Deadlock)에 빠지는 것을 막기 위한 서킷 브레이커(Max Turns) 로직은 현 단계 구현이 다소 복잡하므로 사용자 피로도를 고려하여 우선순위에서 배제함.

* **3.4 정규표현식 이탈에 따른 API 토큰 과금 누수**
  * **최종 합의:** **보류 (패스)**
  * **사유:** 오타(`강사;`, `주이:`) 발생 시 Levenshtein 거리를 계산하여 Fuzzy Matching으로 무료 자동 교정하는 방안은, 현재로선 비용적 타격이나 피로도가 크지 않아 개발 리소스 절약 차원에서 패스함.

## 4. 카드 생성 규칙 및 스케줄링 간섭 리스크

* **4.1 무분별한 카드 증식 방어 (Card Bloat)**
  * **상세 내용:** 문서의 평범한 문장(`~은 ~이다`)을 닥치는 대로 빈칸 카드로 만들거나, 4단계 절차 트리거를 치면 카드가 4장으로 쪼개지면서 사용자가 소화 불가능한 Anki 리뷰 지옥을 초래.
  * **최종 합의:** **승인 및 절차 카드 단일화(Progressive Reveal)**
  * **해결책 상세:** 
    1. **정의 자동 감지 폐지:** 코드가 임의로 카드를 만들지 않게 삭제. 반드시 `[정의]` 등 명시적인 태그를 달았을 때만 카드가 생성되도록 규칙 엄격화.
    2. **점진적 공개 카드:** 4단계 `절차:` 트리거 사용 시 안키 카드를 단 1장만 생성함. 단, 안키 카드 뒷면의 Javascript 템플릿을 수정하여, 사용자가 화면을 한 번 터치할 때마다 1단계, 2단계 텍스트가 순차적으로 부드럽게 나타나게 구현함. 복습 횟수는 1번으로 유지하며 순서 암기 효과는 극대화.

* **4.2 스케줄러 에이전트의 FSRS 알고리즘 훼손**
  * **상세 내용:** 사용자가 에이전트에게 스케줄 조정을 요구했을 때, 에이전트가 SQLite 상의 `due`(복습 예정일)를 강제로 밀어버리면 FSRS 망각 곡선 알고리즘이 "기억력이 엄청나게 유지된 것"으로 착각하여 다음 복습 주기를 기하급수적으로 뻥튀기(Interval Explosion)함.
  * **최종 합의:** **승인 및 Postpone API 전용 연동**
  * **해결책 상세:** 에이전트는 절대 DB의 원시 데이터(날짜)를 직접 조작할 수 없도록 권한을 박탈당함. 대신 Anki 공식 생태계의 **'FSRS Helper 연기(Postpone) 기능' API**만 호출하도록 허용됨. 이를 통해 카드가 며칠 뒤로 밀리더라도, 알고리즘이 수학적 페널티를 보정 분산시켜 장기 기억 곡선의 파괴를 막아냄.

* **4.3 이미지 가림(Image Occlusion)의 기술적 한계**
  * **상세 내용:** MarginNote 코멘트에 `가림:` 이라고만 치면 파이썬 스크립트(컴퓨터 비전 부재)가 이미지 내의 어떤 좌표(x, y)를 박스로 가려야 할지 알 수 없다는 한계.
  * **최종 합의:** **해결 및 당일 수동 워크플로우 정립**
  * **해결책 상세:** 컴퓨터 비전으로 좌표를 강제 추적하는 오버엔지니어링(Over-engineering)을 폐기함. 과거의 원래 훌륭한 설계 의도를 복원하여, 파이썬은 그저 마진노트 이미지를 넣은 **가림 전용 뼈대 카드**만 Anki로 넘김. 
  사용자는 강의를 수강한 **당일**, Anki Desktop을 켜서 Anki 플러그인(Image Occlusion Enhanced)을 통해 마우스로 빨간 박스들을 직접 치는 작업을 정규 스케줄로 편입함. 박스를 친 즉시 카드가 FSRS 스케줄링에 정상 진입함.

## 5. 시스템 UX 및 인지적 과부하 리스크

* **5.1 탭(Tab) 들여쓰기에 대한 과도한 파싱 의존성**
  * **상세 내용:** Logseq에서 사용자가 탭이나 스페이스바를 한 번 실수하면 2차 연습서 문제의 부모(공통자료)-자식(물음) 구조가 와해됨.
  * **최종 합의:** **해결 완료 (사용자 탭핑 불필요)**
  * **해결책 상세:** 사용자가 MarginNote 앱 내에서 문제집을 마인드맵으로 묶어두면, 그 자체로 DB(`ZGROUPNOTEID`)에 완벽한 부모-자식 위계 구조가 저장됨. 파이썬 스크립트는 Logseq 텍스트 파일의 탭을 분석하는 것이 아니라, 마진노트의 본원적 계층 구조를 최우선 진실(Source of Truth)로 삼아 읽어옵니다. 그 결과 컴퓨터가 알아서 들여쓰기 된 Logseq 마크다운 파일을 자동 생성해주므로 사용자는 탭을 칠 필요가 전혀 없습니다.

* **5.2 목차 암기 및 호명의 인지적 스트레스**
  * **상세 내용:** 사용자가 연습서에서 틀린 개념을 헤르메스와 연결할 때, "헤르메스, 1.3.2 문제랑 1.1.1 연결해"라며 기본서 목차 번호를 암기하고 타이핑해야 하는 극심한 인지적 스트레스 발생.
  * **최종 합의:** **승인 및 자동 추천/승인 UX 도입**
  * **해결책 상세:** 목차 번호를 외울 필요를 제거함. 에이전트는 연습서 문제에 포함된 원본 텍스트("상각원가", "법인세" 등 핵심 키워드)를 추출해 교재 목차 DB와 벡터 검색(유사도 매칭)을 수행함.
  이후 슬랙 봇에 **"🤖 힌트: 이 문제는 [6.3.2 당기손익-공정가치측정] 개념과 가장 유사합니다. 연결할까요?"** 와 함께 상위 3개의 후보 버튼을 띄움. 사용자는 버튼 클릭만으로 연결을 확정.
  만약 정답이 후보군에 없다면, **`[🔍 직접 찾기]`** 버튼을 추가로 두어 에이전트에게 "자본유지 접근법 찾아줘" 라고 명령해 손쉽게 연결을 수행(Fallback).

* **5.3 지나치게 복잡한 트리거 규칙 제약 (주객전도 현상)**
  * **상세 내용:** 사용자가 회계학 공부보다 9가지 트리거 규칙(`주의:`, `강사:`)을 외우고 코멘트에 타이핑하는 데 귀중한 에너지를 소모함.
  * **최종 합의:** **승인 및 색상 기반 초격차 단순화 달성**
  * **해결책 상세:** 마진노트 SQLite DB(`ZBOOKNOTE` 테이블) 내에 색상 정보 컬럼(`ZCOLOR`)이 인덱스 숫자로 명확히 저장된다는 사실을 교차 검증을 통해 확인함.
  따라서 수동 텍스트 타이핑을 폐지하고 가장 직관적인 UX를 도입. 파이썬 스크립트에 색상 맵핑 코드를 추가하여, **"빨간색(Index 2) 노드 = 무조건 `⚠주의` 카드 생성", "초록색(Index 3) 노드 = 무조건 `📋절차` 카드 생성"** 방식으로 자동화함. 사용자는 글씨를 타이핑할 필요 없이 터치/펜슬로 노드 색깔만 칠해두면 시스템이 알아서 카드의 종류와 트리거를 분류해 생성합니다.
