# CPA 학습 자동화 시스템 — 팀 개발 마스터플랜 V2 (최종 개정판)

> **💡 [문서 개요]**
> 본 문서는 시스템의 각 모듈(MarginNote, Python, Anki 등)이 독립적으로 포괄하는 기술 스펙과 개발 범위를 선언하고, 이들이 최종적으로 어떻게 파이프라인으로 결합되는지를 정의하는 **팀 업무 분장(WBS)용 아키텍처 설계서**입니다.
> 
> 구버전의 수작업 텍스트 트리거는 전면 폐기되었으며, **"색상 기반 자동화"**, **"안키 스케줄링 훼손 방어"**, **"목차 자동 매핑 UX"** 등 고도화된 아키텍처가 기본 스펙으로 통합되었습니다.

## 1. 독립 컴포넌트 아키텍처 및 업무 분장 (Team WBS)

각 모듈이 파이프라인으로 합쳐지기 전, 어떤 고유의 기술 스펙과 기능들을 품고 개발되어야 하는지 독립된 컴포넌트 단위로 분해한 구조도입니다. 업무 분장(WBS)의 핵심 지표가 됩니다.

```mermaid
mindmap
  root((CPA V2 System))
    (1) 데이터 입력 및 UI 파트<br/>MarginNote Team
      3대 교재 뼈대 해체
      ZCOLOR 색상 매핑 룰
      ZGROUPNOTEID 위계 트리 구축
    (2) 백엔드 코어 파트<br/>Python Team
      PRAGMA 스키마 무결성 검증기
      Frontmatter 마크다운 파서
      AnkiConnect API 연동 브릿지
      5TB 원자적 백업 스크립트
    (3) 프론트엔드 관제 파트<br/>Logseq & Obsidian Team
      2차 연습서 아웃라이너 뷰 세팅
      Dataview 약점 랭킹 대시보드
    (4) 복습 및 스케줄링 파트<br/>Anki System Team

    (5) 메타인지 및 AI 파트<br/>AI Agent Team
      STT 오염 정제용 Custom Dict
      Vector Search 목차 추천 봇
```

### 🛠️ 컴포넌트별 개발 스펙 명세 (업무 분장)

**[1] MarginNote Team (데이터 창고 및 1차 UI 제어)**
* **입력부 설계:** 기본서(TOC 목차), 1차 객관식, 2차 연습서 PDF를 구조적으로 해체하여 마진노트에 Import 가능한 형태로 만듭니다. (특히 기본서 목차는 `TOC_Mapping.json`으로 별도 추출하여 개정판 출시에 대응합니다).
* **트리거 및 위계 시스템 구축:** 마진노트 앱 내부의 색상(`ZCOLOR`)을 카드의 종류(함정, 절차, 강사팁)를 결정하는 유일한 트리거로 매핑합니다. 또한 사용자의 마인드맵 묶음 동작(`ZGROUPNOTEID`)이 자동으로 위계가 되도록 데이터베이스 룰을 세팅합니다.

**[2] Python Team (동기화 엔진 및 백엔드 코어)**
* **파서 및 생성기 개발:** 마진노트의 `.marginnotes` SQLite를 읽어들여 Logseq/Obsidian용 마크다운(프론트매터 포함)을 동적으로 생성하는 스크립트를 작성합니다.
* **무결성 및 안정성 모듈:** 읽기 전 `PRAGMA table_info`를 통해 마진노트 스키마 붕괴를 검증하고, Anki 포트(8765)에 핑을 날려 앱이 꺼져있을 경우 백그라운드로 강제 실행시키는 Health Check 로직을 개발합니다.
* **백업 스크립트:** 마진노트, 옵시디언, 안키 DB를 한 날 한 시에 묶어 단일 ZIP 파일로 구글 드라이브(5TB)에 올리는 원자적 백업 스크립트를 작성합니다.

**[3] Logseq & Obsidian Team (계층/메타인지 프론트엔드)**
* **Logseq 환경 구성:** 파이썬이 던져주는 마크다운 구조체를 읽어, 2차 연습서의 복잡한 [공통자료]-[물음1,2]-[해설] 위계를 직관적인 아웃라이너(Outliner) 형태로 랜더링하도록 세팅합니다.
* **Obsidian 대시보드:** Dataview 플러그인 쿼리를 작성하여, 안키의 Revlog 테이블에서 발생한 '오답 횟수'를 취합해 "8장 자산손상 - 27회 틀림"과 같이 실시간 약점 랭킹 보드를 구축합니다.

**[4] Anki System Team (복습 및 스케줄링)**
* **카드 워크플로우 통제:** 파이썬으로부터 넘어온 '이미지 뼈대 카드'에 사용자가 당일 수동으로 빨간 박스를 치도록 Image Occlusion Enhanced 플러그인 워크플로우를 확립합니다.
* **모바일 단방향 락(Lock) 정책:** 안드로이드 폰(AnkiDroid)에서 수정한 데이터가 덮어씌워져 날아가는 것을 막기 위해, 모바일에선 절대 카드를 수정하지 않는 '단방향 읽기 전용' 룰을 시스템에 정착시킵니다.

**[5] AI Agent Team (LLM 지식 베이스 및 봇 제어)**
* **STT 환각 필터링:** 강의 녹음이 ChromaDB에 들어가기 전, 마진노트에서 추출된 '사용자 전용 용어 사전(Custom Dict)'을 바탕으로 소형 LLM 필터가 "자본유지 젓갈법" 같은 오탈자를 정제하도록 파이프라인을 구축합니다.
* **Vector Search 추천 봇:** 사용자가 Slack을 통해 연습서 문제의 연결을 지시하면, 관련 교재 목차를 찾아 원클릭으로 맵핑해주는 에이전트를 개발합니다.
* **FSRS 스케줄 보호 API:** 에이전트가 임의로 안키 DB의 `due` 날짜를 조작하지 못하게 막고, 오직 정식 `FSRS Postpone API`만을 호출하도록 권한을 락(Lock)합니다.

## 2. 통합 엔드투엔드(E2E) 파이프라인

위 5개 팀이 개발한 고유 컴포넌트들이 실제 사용자 환경에서 **어떤 순서(1단계~7단계)로 결합되어 데이터 순환 고리(Circular Workflow)를 만들어내는지** 보여주는 최종 통합 파이프라인입니다.

```mermaid
graph TD

    subgraph S1 ["[1] 지식 & 뼈대 파이프라인"]
        A1[기본서 PDF] -->|TOC_Mapping.json 추출| B[MarginNote 4 뼈대 Import]
        A2[1차 객관식 문제집] -->|문항/해설 슬라이싱| B
```

### 2.1 E2E 파이프라인 작동 원리
1. **[1] 파이프라인:** 3대 교재와 STT 스크립트가 파싱/정제되어 MarginNote와 지식베이스(ChromaDB)에 각각 적재됩니다.
2. **[2] 마진노트 제어:** 사용자가 마진노트 UI에서 색상(`ZCOLOR`)을 칠하고 노드를 묶으면(`ZGROUPNOTEID`), 이 행위 자체가 코딩 룰이 되어 DB에 저장됩니다. (수동 텍스트 트리거 완전 폐지).
3. **[3] 파이썬 엔진:** 백엔드가 이 SQLite를 안전하게(PRAGMA 검증) 추출하여 마크다운 파일들을 쏟아내고 Anki API로 데이터를 푸시합니다.
4. **[4] 프론트엔드 관제:** 쏟아진 마크다운을 Logseq가 받아 2차 연습서의 복잡한 계층을 아웃라이너로 렌더링하고, Obsidian은 약점 대시보드를 시각화합니다.
5. **[5] 안키 복습:** Anki로 넘어온 데이터 중 이미지는 사용자가 당일 직접 가림 박스를 치며, 모바일(AnkiDroid)에서는 오직 복습(읽기)만 수행하여 통계를 누적시킵니다.
6. **[6] AI 관제:** 누적된 통계와 틀린 문제를 바탕으로 에이전트가 복습 스케줄을 연기(Postpone API)하거나 목차 연결망을 찾아냅니다.
7. **[7] 원자적 백업:** 위 모든 파이프라인의 핵심 폴더 3곳이 동일한 시점에 하나의 단일 압축 파일로 클라우드에 영구 백업됩니다.

## 3. 마진노트(MarginNote4) 데이터베이스 코어 아키텍처

스크립트가 직접 읽어들이는 마진노트의 SQLite `.marginnotes` 테이블 명세입니다. 앱 크래시 방지를 위해 라이브 DB를 직접 건드리지 않고 항상 `/tmp/` 폴더로 스냅샷 복사된 DB를 읽어옵니다.

### 3.1 🚨 비공식 DB 파싱 취약성 및 무결성 검증 (Sanity Check)
앱 메이저 업데이트로 테이블 스키마가 약간만 변경되어도 스크립트에 치명적 에러가 발생합니다.
* **해결책:** Python Team은 스크립트 최상단에 `PRAGMA table_info(ZBOOKNOTE)` 명령을 실행하여 필수 컬럼(`ZNOTEID`, `ZCOLOR`, `ZGROUPNOTEID`)의 존재 여부를 1차 검사하도록 강제됩니다. 변경 감지 시 즉각 슬랙 알람을 송출합니다.

### 3.2 핵심 테이블 명세
| 테이블 / 컬럼 | 내용 및 활용 용도 |
|--------------|----------------|
| `ZBOOKNOTE.ZNOTEID` | 노드 고유 ID (UUID). Anki 카드 중복 방지 및 업데이트에 사용됨. |
| `ZBOOKNOTE.ZGROUPNOTEID` | **[✨ 핵심]** 부모 노드 ID. 이를 통해 2차 연습서의 다중 위계를 완벽히 파악하여 Logseq 마크다운 들여쓰기를 자동 파싱합니다. |
| `ZBOOKNOTE.ZCOLOR` | **[✨ 핵심]** 카드의 종류를 결정하는 색상 인덱스(예: 2=빨강, 3=초록). 수동 텍스트 트리거 없이 시스템을 자동 제어합니다. |

## 4. 🎨 색상 기반 초격차 UX: 카드 생성 트리거 (Trigger Rules)

과거 수험생이 코멘트 창에 `주의:`, `절차:` 등 9가지 텍스트를 일일이 타이핑하며 인지적 에너지를 소모했던 현상을 완벽히 폐기했습니다. **오직 터치/펜슬로 지정한 노드 색깔(`ZCOLOR`)만이 카드의 본질을 결정합니다.**

*(※ 무분별한 카드가 양산되는 것을 막기 위해, 과거의 '문장 자동 빈칸 변환 규칙'은 모두 폐기되었습니다. 카드 생성은 철저히 사용자의 의도(색상)에 종속됩니다.)*

### 4.1 색상 인덱스 매핑 룰

| 노드 색상 (ZCOLOR) | 생성되는 Anki 카드 유형 | 프론트/백엔드 작동 로직 | SRS 스케줄 |
|-----------------|----------------------|-----------------------|----------|
| **🔴 빨강 (Index 2)** | **⚠ [주의/함정 카드]** | 앞면: "⚠ 주의할 함정은?"<br>뒷면: 사용자가 하이라이트한 텍스트/이미지 노출 | 짧음 |
| **🔵 파랑 (Index 4)** | **🎤 [강사 팁 카드]** | 앞면: "🎤 강사의 실무/보충 팁"<br>뒷면: 팁 내용 노출 | 표준 |
| **🟢 초록 (Index 3)** | **📋 [절차/단계 카드]** | **[점진적 공개 (Progressive Reveal) 단일 카드]**<br>과거 4단계 절차를 4장의 카드로 쪼개던 방식을 버림. 카드는 **단 1장**만 생성. 대신 Anki 뒷면 JS 템플릿이 화면 터치 시마다 1단계, 2단계를 순차적으로 보여줌. 리뷰 지옥 방어 완료. | 짧음 |
| 기타 색상 (기본형) | 일반 Q&A 카드 | 명시적인 색상이 없을 시 가장 평범한 Q/A 모델로 생성 | 표준 |

### 4.2 가림 카드 (Image Occlusion) 수동 워크플로우
* **기술적 한계:** 과거 파이썬 코드가 코멘트의 `가림:` 텍스트를 보고 화면의 어느 (X, Y) 좌표를 가려야 할지 알 수 없는 한계가 존재했습니다.
* **최종 합의 워크플로우:** 파이썬은 마진노트의 이미지 캡처본을 담은 **'가림 뼈대 카드'**만 Anki로 보냅니다. 사용자는 강의를 들은 **당일**, Anki Desktop에서 플러그인을 이용해 수동으로 빨간 박스를 치는 것을 정규 스케줄로 편입시킵니다. 수동 작업 직후 FSRS 스케줄에 진입합니다.

## 5. 시스템 안전 수칙 및 관리자 룰 (절대 엄수)

통합 파이프라인이 크래시나 무한 증식 없이 영구 운영되기 위한 최후의 방어 수칙입니다.

1. **카드 수동 복제/분할 영구 금지 (UUID 파괴 방지):**
   * 시스템은 카드 중복 생성을 막기 위해 MarginNote의 `ZNOTEID`를 Anki 뒷면에 심어 둡니다.
