# CPA 학습 자동화 시스템 — 궁극의 통합 인수인계 매뉴얼 (Ultimate Handover Masterplan)

> **💡 [문서 개요]**
> 본 문서는 기존에 작성되었던 시스템 마스터플랜(초안~최종본), 하위 연동 매뉴얼(카드 생성규칙, 통합 시스템 매뉴얼), 에이전트 확장팩, 시스템 한계점 및 개선안, 그리고 시스템 설치 및 검증 결과(Phase 0 완료 매뉴얼)를 단 하나의 내용 유실도 없이 완벽하게 병합한 **궁극의 단일 진실 공급원(Single Source of Truth)**입니다.
> 처음 시스템을 접하는 담당자가 코어 아키텍처부터 구체적인 DB 테이블, 안키 스케줄링 분 단위 설정, 파이썬 코드 예시, 그리고 최신 AI 에이전트 연동 논리까지 완벽하게 인수인계받을 수 있도록, 참조문서 내의 모든 예시와 데이터를 그대로 보존하여 가장 상세하게 작성하였습니다.

## 1. 프로젝트 개요 및 4대 코어 아키텍처 

CPA 학습 시스템은 수작업 암기를 넘어, **"내가 무엇을 알고 모르는지(메타인지)"를 완벽하게 통제**하고, 학습 효율을 극대화하기 위한 자율형 AI 기반의 사이보그 컨설턴트 환경입니다. 
데이터는 '구조 뼈대 생성'부터 '약점 역추적'까지 완벽한 순환 고리(Circular Workflow)를 형성합니다.

### 1.1 시스템 전체 구조도 및 파이프라인

```mermaid
graph TD
    A[기본서 TOC 및 문항 자르기 시스템] -->|아웃라이너 뼈대 OPML 생성| B[MarginNote 4 역삽입 Import]
    B -->|사용자 학습: 선 긋기로 오답 원인 연결| C[MarginNote SQLite DB]
    C -->|Python 1차 변환| D[Logseq & Obsidian 마크다운 Vault]
    D -->|Python 2차 변환| E[Anki 자동 복습기]
    E -->|오답(Lapse) 통계 발생| F[Obsidian 대시보드 랭킹 산출]
    F -->|약점 역추적 피드백| E
```

**[기본 데이터 파이프라인]**
```text
MarginNote4     →     공유 Markdown 폴더          →     Anki
(창고)                 ↙      ↓       ↘               (복습)
                Obsidian   Logseq    AI 에이전트       Galaxy에서
                그래프뷰   아웃라이너  LLM Wiki         맞음/틀림만
```

### 1.2 시스템별 상세 역할 분담

| 시스템 | 역할 (기능) | 상세 설명 |
|--------|-----------|---------|
| **MarginNote4** | **창고 & 입력단** | 교재, 필기, 오답이 모이는 원천 DB. PDF 발췌, 트리거 코멘트 작성. 각 노드는 고유 URL(`marginnote4app://`)을 가집니다. 시각적 마인드맵 선 긋기로 '암묵지'와 '약점'을 메타인지적으로 기록합니다. |
| **Logseq** | **계층 통제실** | 숲의 '나무' 관리. 2차 연습서의 복잡한 [공통자료]-[물음1,2] 위계를 마크다운 **들여쓰기(Tab)**로 직관적으로 통제하는 아웃라이너입니다. |
| **Obsidian** | **메타인지 관제실** | 숲 '전체' 관리. Anki 오답 통계와 원인 연결망을 취합해 "내가 어떤 개념 때문에 몇 번 틀렸는지" 실시간 약점 랭킹 제공 및 지식 그래프 뷰 기능을 제공합니다. |
| **Anki** | **자동 복습기** | 쪼개진 덱 없이 FSRS 망각곡선이 단일 덱에서 100% 스케줄을 통제합니다. 딥링크 버튼으로 원문/맥락 이동 기능을 제공합니다. 모바일(Galaxy)에서는 복습만 수행합니다. |
| **AI (Hermes 등)** | **LLM 지식 베이스** | 공유 폴더의 Markdown을 읽고 Q&A, 스케줄 조정, 환각 팩트체크 등 다중 에이전트 컨설턴트 역할을 수행합니다. |

## 3. 마진노트(MarginNote4) 데이터베이스 아키텍처 상세

스크립트가 직접 읽어들이는 마진노트의 SQLite `.marginnotes` (`.marginpkg` ZIP 압축 해제본) 테이블 명세입니다. 마진노트 앱 크래시 방지를 위해 라이브 DB를 직접 건드리지 않고, 파이썬은 항상 `/tmp/` 폴더로 스냅샷 복사된 DB나 `.marginpkg` 백업본을 읽어옵니다.

### 3.1 ZBOOKNOTE (노드 테이블)
| 컬럼 | 내용 |
|------|------|
| `ZNOTEID` | 노드 고유 ID (UUID). Anki 카드 중복 방지(멱등성)에 사용됨. |
| `ZNOTETITLE` | 노드 제목 |
| `ZHIGHLIGHT_TEXT` | PDF 발췌 텍스트 |
| `ZGROUPNOTEID` | 부모 노드 ID (계층 구조) |
| `ZTOPICID` | 소속 노트북/스터디셋 ID |
| `ZTYPE` | 노드 타입 (7=일반, 256=마인드맵, 3=그룹, 4=기타) |
| `ZZINDEX` | 정렬 순서 |
| `ZMINDLINKS` | 자식 노드들의 ID 리스트 (`|`로 구분). 계층 구조의 '부모-자식' 관계 추적에 사용. |
| `ZNOTES_TEXT` | 노트 텍스트 |
| `ZNOTES` | 노트 (blob) |

### 3.2 ZBOOKCOMMENT (코멘트 테이블) - 트리거 파싱의 핵심
| 컬럼 | 내용 |
|------|------|
| `ZCOMMENTID` | 코멘트 고유 ID |
| `ZCOMMENTTEXT` | **코멘트 텍스트 (트리거 파싱 대상). 첫 줄의 `주의:`, `강사:` 등을 인식.** |
| `ZMARKNOTEID` | 연결된 부모 노드의 ZNOTEID |
| `ZTOPICID` | 소속 노트북 ID |
| `ZDATE` | 작성 시각 (Apple epoch) |

### 3.3 ZTOPIC (노트북/스터디셋 테이블)
| 컬럼 | 내용 |
|------|------|
| `ZTOPICID` | 노트북 고유 ID |
| `ZTITLE` | 노트북 이름 |

## 4. Phase별 초정밀 시스템 실행 계획 (Micro-Execution Plan)

### [Phase 0] 교재 3대 아키텍처 자동화 및 TOC 파이프라인
기본서와 문제집을 통째로 보지 않고 뼈대 단위로 해체하여 MarginNote와 Logseq에 Import합니다.
* **Task 0.1 [PDF Parser]:** 기본서 PDF 북마크 추출 및 JSON 파싱 (OPML 생성).
* **Task 0.2 [1차/2차 Slicer]:** 객관식 문제집(문제+해설), 연습서 위계(공통자료->물음->해설) 정밀 크롭.
* **Task 0.3 [OPML Generator]:** 마진노트/Logseq Import용 뼈대 파일 생성.
* **✨ [핵심 개선] 목차(TOC)-URL 인덱스 DB 구축:** 마진노트로 임포트된 목차(예: `1.1.1 자산손상`)와 마진노트의 고유 URL(`marginnote4app://...`)을 파이썬이 미리 1:1 매핑하여 DB로 구축. (오답 발생 시 수동 복사 대신 에이전트가 번호만으로 자동 연결망을 엮기 위함)

### [Phase 1~2] DB 추출 및 마크다운 변환 엔진
* **Task 1.1~1.5 [MD Writer 및 Media Exporter]:** SQLite 데이터 파싱 후 선 연결망 추출 및 프론트매터가 포함된 Markdown 생성.
* **✨ [핵심 개선] Logseq 다중 위계 파싱 방어:**
  * **예상 문제점:** 2차 연습서의 날것 구조(`공통자료 → 물음1 → 해설`)가 그대로 Anki로 가면 부모 맥락을 상실.
  * **해결책:** 추출된 날것 텍스트를 사용자가 Logseq에서 **탭(Tab) 키**로 들여쓰기합니다.
    ```markdown
    - [문제 1 공통 자료]
      - 물음 1. (가) 계산하시오.
      - 물음 2. 분개하시오.
    ```
  * 파이썬 스크립트는 이 Logseq의 **마크다운 들여쓰기(Indentation)**를 읽어 부모를 파악하고, `물음 1` Anki 카드 뒷면에 공통자료 내용을 자동 주입합니다.
* **✨ [핵심 개선] WebP 자동 압축:** 이미지 복사 전 해상도를 모바일 친화적으로 축소하고 `.webp` 포맷으로 강제 변환하여 모바일 동기화 지연과 용량 폭발을 80% 이상 축소합니다.

### [Phase 3~4] Anki 동기화, 딥링크 브릿지 및 강의 RAG
* **Task 3.1~3.4 [DeepLink & Sync Controller]:** 카드 뒷면에 `[ M ]`(마진노트 원문), `[ L ]`(Logseq 맥락), `[ O ]`(Obsidian 그래프) 딥링크 주입. 변경된 노트만 동기화.
* **✨ [핵심 개선] Regex vs 에이전트 하이브리드 파싱:** 
  * 오타 없이 트리거를 쓰면 파이썬 정규표현식으로 무료 생성.
  * 모바일의 대충 작성된 메모나 비정형 데이터는 Hermes LLM으로 보내 JSON 규격화 파싱.
* **단방향 덮어쓰기 방어 (Inbox 룰):** 스마트폰(Galaxy AnkiDroid)에서 수정한 내용은 동기화 시 덮어씌워져 날아갑니다. 따라서 폰에서는 조회/학습만 하고, 카드 수정은 Slack/Telegram 봇(Inbox)에 보내 Mac에서 일괄 처리합니다.
* **Task 4.1~4.3 [강의 텍스트 RAG 파이프라인]:** `.srt` / `.txt` 새 강의 스크립트 감지 시, 소형 LLM(GPT-4o-mini 등)으로 회계 전문 용어 오탈자를 1차 교정하고 `SemanticChunker`로 분절해 로컬 ChromaDB에 적재합니다. ("중요합니다" 발언 감지 시 해당 개념 전체 노드에 빈출 태그 자동 부여)

### [Phase 5] 카파시(Karpathy) LLM OS와 헤르메스 에이전트 확장
* **Task 5.1~5.3 [Memory Editor (RAM -> Disk 아키텍처)]:** 
  * 1만 토큰의 긴 대화를 매번 읽히지 않기 위해 역할을 분리합니다.
  * **Memory Editor 에이전트:** 세션 종료 시 슬랙 대화 로그를 모니터링하다 핵심 정수(약점 등)만 뽑아 하드디스크(`LLM_Wiki/User_State.md`)에 단 한 줄로 "던져넣고" 사라집니다. 튜터는 통신 전 이 RAM만 로드합니다. (원본 대화는 HTML `<details>` 태그 내에 숨겨 보존)
* **Task 5.4~5.5 [Tutor & Checker 에이전트 팀 (이기종 교차 검증)]:** 
  * 구독 중인 OpenCode Go를 활용합니다.
  * **Tutor Agent:** 논리력이 뛰어난 `Claude 3.5 Sonnet` 할당. (강의 RAG 및 웹 검색, 답변 생성)
  * **Checker Agent:** 지식 베이스 팩트체크가 강한 `Qwen-Max` 또는 `Gemini 1.5 Pro` 할당. 서로 다른 두 모델이 상호 감시하여 회계/세법 환각(Hallucination)을 원천 차단. Checker는 반드시 IFRS 기준서 등 문단 번호를 색인해야 합니다.
* **Task 5.6 [Scheduler Agent (스마트 푸시)]:** 
  * 일상(저녁 회식 등) 캘린더 모니터링 후 Anki 듀데이트 물리적 승인/조정.
  * **✨ [핵심 개선] 미니 데일리 덱(Daily Deck) 푸시:** 전체 모바일 동기화의 딜레이를 없애고자, 에이전트가 그 날 복습할 카드만 뽑아 가벼운 미니 덱을 모바일로 푸시합니다.

### [Phase 6] 무제한 스냅샷 백업 시스템 (안전망)
* **Task 6.1~6.2 [Unlimited Archiving]:** 5TB 구글 드라이브 용량을 활용해, 에이전트가 DB나 마크다운을 수정하기 전 무조건 전체 폴더를 `YYYYMMDD_HHMMSS.zip` 형태로 스냅샷 압축합니다. 특정 시점으로의 완벽한 롤백이 보장되며, 대규모 수정(Suspend 등)은 Slack에 승인(Approve/Reject) 버튼을 송출해 하이브리드 결재 시스템을 거칩니다.

## 5. 카드 생성 규칙, 데이터 허용 형태 및 상세 트리거 (Trigger Rules)

### 5.1 카드에 들어갈 수 있는 데이터 형태
모든 트리거에서 공통 지원됩니다.
| 입력 데이터 형태 | Anki 카드 반영 형태 | 비고 |
|----------------|------------------|------|
| 텍스트만 | 텍스트 (Markdown 렌더링) | |
| 이미지만 | 이미지 원본 | 비교표, 재무제표 양식, 강사 판서 등 |
| 텍스트 다수 | 순서대로 세로 나열 | |
| 이미지 다수 | 순서대로 세로 나열 | 여러 장 풀이 통합 |
| 텍스트+이미지 | 텍스트 위, 이미지 아래 | 문제 텍스트 + 풀이 캡처 형태 |

### 5.2 기본형 카드 (트리거 없음)
코멘트에 특별한 키워드가 없으면 일반 Q&A 카드입니다.
- 앞면: 제목 / 뒷면: 내용. (SRS: 표준 간격)

#### 💡 정의(Definition) 자동 감지 패턴
내용 중 아래 패턴이 감지되면 빈칸(Cloze) 카드 1장을 추가로 자동 생성합니다.
* `X이란 Y이다`, `X은/는 Y을/를 말한다`, `X은/는 Y을/를 의미한다`, `X(이)라 함은 Y`, `X: Y` (콜론)

### 5.3 9대 특수 트리거 상세

| 트리거 | 텍스트 입력 예시 | 생성 원리 (앞면/뒷면 구조) | SRS 스케줄링 |

## 6. 메타인지 추적 및 약점 연결망 구축 원리 (2차 연습서 대응)

2차 연습서 문제는 복합 개념(예: 자산손상+법인세)이 등장하므로, 사용자가 오답의 근본 원인을 연결하는 과정이 필수적입니다. 이 방식은 고통스러운 수동 입력에서 완전 자동화로 진화했습니다.

### 6.1 오답 연결망 시스템의 진화
* **1세대 (텍스트 직접 입력):** 코멘트에 `오답: 2차 p.127 q2. 법인세 효과 놓침 / 연결: 8.1, 12.3` 타이핑.
* **2세대 (Spaghetti Linking):** 마인드맵 상에서 마우스로 반대편 끝 노드까지 길게 점선을 긋기. (노드가 많아지면 물리적으로 불가능)
* **3세대 (최종 완성본 - 헤르메스 TOC 자동 매핑):**
  Phase 0에서 구축된 목차-URL 매핑 DB를 활용. 사용자는 오답 발생 시 Slack/Telegram 헤르메스 에이전트에게 이렇게 명령합니다.
  > 🗣️ "헤르메스, 1.3.2 문제(연습서)랑 1.1.1(자산손상 목차) 연결해"
  에이전트가 고유 URL을 뽑아 백그라운드 마크다운(Frontmatter) 및 Anki Hidden Tag에 `#약점::자산손상`, `#약점::법인세` 형태로 하드 링크를 생성.

### 6.2 궁극의 오답 통계 (진실의 방 대시보드)
단순한 링크를 넘어, Anki와 Obsidian이 연동하여 완벽한 메타인지를 제공합니다.
1. **진실의 방 (Anki Revlog 통계):** 스마트폰 AnkiDroid에서 사용자가 `[다시(Again)]`를 누른 실제 누적 오답 횟수(Lapses)를 파이썬 스크립트가 SQLite에서 긁어옴.
2. **통합 시각화 (Obsidian Dataview 대시보드):** Anki의 객관적 통계와 앞서 구축한 약점 연결망을 취합하여 실시간 랭킹 산출.
   > 📊 **나의 2차 연습서 약점 랭킹 (실시간 업데이트)**
   > 1. `[[8장 자산손상]]` : 연관 문제 도합 27번 틀림 (치명적 약점 경고)
   > 2. `[[12장 법인세]]` : 연관 문제 도합 15번 틀림
   > 3. `[[5장 리스]]` : 연관 문제 도합 3번 틀림

## 7. 시스템 방어망 및 관리자 상세 (Loopholes Mitigations & Admin)

통합 시스템이 크래시 없이 안정적으로 영구 운영되기 위한 3대 안전 기제입니다.

### 7.1 카드 중복 방지 및 멱등성 (Idempotency)
Anki 카드 내부에 마진노트의 고유 `ZNOTEID` (UUID) 필드를 히든으로 박아 넣습니다. 마진노트에서 오타를 고치고 스크립트를 수백 번 반복 추출해도, 카드가 무한 증식하지 않으며 기존 카드의 내용만 스마트하게 덮어쓰기(업데이트) 됩니다.

### 7.2 태그 트리 파손 방지
`8.1 자산 손상` 등 띄어쓰기나 특수문자가 포함된 개념명이 오답 태그로 넘어갈 경우 Anki의 태그 체계가 무너집니다. 따라서 스크립트가 사전에 정규식으로 `#약점::8.1_자산_손상` 형식의 언더스코어로 강제 치환 및 정제하여 삽입합니다.

### 7.3 데드 링크 폴백 (Fallback URL)
Logseq/Obsidian 마크다운 파일 동기화가 깨져 로컬 딥링크(`obsidian://open...`)가 죽어 맥락이 끊기더라도, 모든 Anki 카드 뒷면 하단에는 `marginnote4app://...` 원문 PDF 고유 URL이 박혀 있어, 언제든 터치하여 교재 원본 화면으로 살아 돌아갈 수 있습니다.

### 7.4 AnkiConnect 시스템 연동 예시 (API Usage)
실제 시스템 통신을 점검하고 검증하기 위한 API 사용법 요약입니다.
* **서버 상태 확인:**
  ```bash
  curl -s http://localhost:8765 -X POST -H "Content-Type: application/json" -d '{"action":"version","version":6}'
  ```
  *(정상 응답: `{"result": 6, "error": null}`)*
* **파이썬 덱 생성 및 카드 추가 API 구조:**
  ```python
  anki("createDeck", deck="CPA::재무관리")
  anki("addNote", note={
      "deckName": "CPA::재무관리",
      "modelName": "Basic",
      "fields": {"Front": "⚠ 주의할 점은?", "Back": "NPV와 PI 결과가 상반되면 반드시 NPV 우선"},
      "tags": ["CPA::재무관리", "트리거::주의"]
  })
  ```

### 7.5 Phase 0 실제 카드 생성 검증 결과 요약
* **테스트 일시:** 2026-06-06 21:52 KST
* **대상:** 재무관리 스터디셋
* **결과:**
  * **[주의]** 맥락: `📍 4 경제성평가 방법간 비교(2)... > 1. NPV 법과 PI 법` / 앞면: `⚠ 주의할 점은?` / 뒷면: `NPV와 PI 결과가 상반되면 반드시 NPV를 우선한다.`
  * **[절차]** 맥락: `📍 ... > 3. 상반되는 의사결정 해결` / 앞면: `📋 절차는?` / 뒷면: `1. 현금흐름 차이(증분) 계산 → 2. NPV 계산 → 3. NPV > 0 (큰 안) → 4. NPV < 0 (작은 안)`
  * **[강사]** 맥락: `📍 4 경제성평가 방법간 비교` / 앞면: `🎤 강사 보충 설명` / 뒷면: `시험에서 상반 문제 나오면, 증분현금흐름으로 풀어도 동일 결과`
* **성공 여부:** 3장 모두 Anki `CPA::재무관리` 덱에 정상 삽입 완료됨.

## 8. 부록 (Appendix): 설계 철학 및 과거 로그 복원본

통합 마스터플랜의 기술적 명세를 뒷받침하는, 사용자님의 '설계 철학' 및 '폐기된 아이디어'를 기록한 아카이브입니다. 인수인계 시 시스템이 "왜 이렇게 설계되었는지"를 이해하는 핵심 배경지식이 됩니다.

### 8.1 멀티 에이전트 체제의 진정한 도입 목적
사용자님께서 다수의 도커(Docker) 기반 에이전트를 도입하고 슬랙(Slack)으로 소통하게 한 것은 단순한 업무 분장이 아닙니다. 과거 대화에서 사용자님이 직접 밝히신 3대 핵심 목적은 다음과 같습니다.

1. **에이전트 간 환각(Hallucination)의 원천 차단:** 
   이기종 모델(Claude vs Qwen/Gemini)이 서로 크로스 체크(Maker & Checker)하도록 만들기 위함입니다.
2. **세션별 텍스트 토큰 한계의 극복:** 
   하나의 에이전트에게 모든 것을 몰아주면 컨텍스트 윈도우가 터지는 문제를 막기 위함입니다.
3. **영구적인 지식 축적 (LLM Wiki 아키텍처):** 
   안드레 카파시(Andrej Karpathy)의 LLM OS 개념을 도입한 근본적 이유입니다. 사용자님은 특정 에이전트(Memory Editor)가 세션의 핵심만 잘 요약해서 뽑아내어 저장소(LLM Wiki)에 계속 던지기만 하도록 역할을 한정시킴으로써, 대화가 증발하지 않고 영구적으로 누적되는 구조를 확립했습니다.

### 8.2 강의 녹음 텍스트 (RAG 파이프라인)의 정확한 출처
강의 텍스트가 어디서 어떻게 생성되는지에 대한 구체적 워크플로우입니다.
* **출처 파이프라인:** 사용자님이 수업 중 작성하는 굿노트(GoodNotes) 필기 및 녹음 데이터가 자동 백업되고, 이를 다른 컴퓨터에서 STT(Speech-to-Text)를 사용하여 `.srt` 및 `.txt` 형식으로 변환하여 생성됩니다.
* **통합 이유:** 회계/재무관리는 글로벌 공통 지식이므로, 에이전트가 이 강의 스크립트와 인터넷 자료, 그리고 마진노트 연결망을 총동원하여 사용자님의 질문에 '전담 튜터'처럼 구체적으로 답변하게 만들기 위함이었습니다. (과거 제안되었던 "Anki 카드 뒷면에 텍스트를 단순 박제하자"는 아이디어는 에이전트의 활용도를 낮추므로 반려되었습니다.)

### 8.3 사용자 피드백으로 '폐기(Reject)'된 시스템 아이디어
다음 기능들은 과거에 모델이 제안했으나 사용자님의 명시적인 판단 하에 폐기된 정책들입니다. 향후 시스템 유지보수나 확장을 담당할 관리자는 이 기능들을 다시 무리하게 도입하지 않아야 합니다.

1. **강제 코드 프리즈 (Code Freeze) 도입 반려:** 
   * **과거 제안:** 시험 100일 전부터는 시스템(파이썬 스크립트 등) 수정을 일절 금지하여 낭비를 막자.
   * **사용자 결정:** 반려 ("이건 필요없다"). 수험생 본인의 판단 하에 자유로운 시스템 유지보수와 최적화를 허용.
2. **약점 격리 병동 (Quarantine) 룰 반려:** 
   * **과거 제안:** 약점 카드는 백지 인출 테스트를 통과하기 전까지 대시보드 알람을 해제하지 못하게 강제하자.
   * **사용자 결정:** 반려 ("일단 이건 패스"). 사용자에게 과도한 스트레스나 제약을 주는 UX는 불필요함.
3. **마진노트 '프록시 노드' 생성 아이디어 완벽 폐기:**
   * **과거 제안:** 장거리 선 긋기 시 마인드맵이 지저분해지는 것(Spaghetti Links)을 막기 위해 가상의 노드(다리)를 만들어 연결하자.
   * **사용자 결정:** 마진노트4의 각 카드가 고유 URL(`marginnote4app://`)을 가지고 있음을 확인하고, 사용자님이 직접 **"헤르메스 에이전트에게 목차 번호만 부르면 알아서 하드 링크를 엮어주는 시스템"**을 고안해 내어 완벽하게 대체/폐기됨.

### 8.4 스마트폰 모바일 UX의 통제 (Inbox 패턴의 당위성)
* **배경:** 스마트폰(Galaxy AnkiDroid)에서 데이터를 수정하고 PC로 가져오려 할 경우, SQLite DB Lock 충돌이나 단방향 덮어쓰기(Override)로 인한 데이터 증발이 발생합니다.
* **해결책의 구체화:** 사용자님은 모바일에서 절대 시스템 원본을 수정하지 않도록 정책을 정했습니다. 그 대신 모바일에서는 '텔레그램 봇이나 슬랙 봇'을 통해 "어디 부분 수정해"라고 텍스트만 보내고(Inbox), 귀가 후 에이전트나 Mac에서 이 내용을 반영하는 방식을 채택했습니다. 이는 데이터 손실을 막는 가장 강력하고 직관적인 하이브리드 패턴입니다.

### 8.5 백업 시스템의 스케일 (5TB 인프라)
* 과거 도커 및 Git을 이용한 저장소 최적화 논의 시, 미디어(이미지/음성) 파일의 용량 문제가 대두되었습니다.
* 그러나 사용자님의 구글 드라이브 용량은 **5TB**로 매우 넉넉하며, 모든 강의 텍스트와 DB 스냅샷이 이미 구글 드라이브에 안전하게 보관되고 있습니다. 따라서 에이전트가 무거운 SQLite 스냅샷 복사본을 주기적으로 찍어내어 보관하더라도 스토리지 걱정 없이 완벽한 롤백이 가능하다는 점이 이 시스템의 숨겨진 장점입니다.

## 9. 시스템 아키텍처 한계점 및 최종 고도화 합의 (Post-Audit Solutions)

> **💡 [Section 9 개요]**
> 아래 내용은 1~8장에 명세된 기존 시스템 아키텍처에 내포된 구조적, 운영적 한계점들을 5대 대분류로 점검하고, 사용자(수험생)와의 심층 논의를 통해 확정된 **최종 해결책(Solution) 및 방어 코드(Mitigation)**를 기록한 최종 고도화 합의서입니다. 향후 파이썬 스크립트 수정 및 에이전트 구축 시 이 9장의 합의를 최우선 기준으로 개발을 진행합니다.

### 9.1 외부 생태계 종속성 및 인프라 가용성 리스크

* **9.1.1 MarginNote URL 스킴 종속성 리스크**
  * **상세 내용:** 모든 오답 연결망과 메타인지 통계가 MarginNote의 고유 URL(`marginnote4app://`)에 기반함. 애플의 샌드박스 정책 변경이나 MarginNote 앱 업데이트 시 URL 스킴 작동이 중단될 위험 존재.
  * **최종 합의:** **수용 (패스)**
  * **사유:** 애플 생태계의 폐쇄성과 앱 종속성은 어쩔 수 없는 한계로 인정하며, 현재 환경에서 가장 효율적인 방식이므로 리스크를 안고 감.

* **9.1.2 비공식 DB 파싱 취약성 (스키마 변경 대비)**
  * **상세 내용:** MarginNote 메이저 업데이트 시 내부 `.marginnotes` SQLite 테이블 스키마(예: `ZBOOKNOTE`, `ZBOOKCOMMENT`)가 약간만 변경되어도 파이썬 추출 스크립트에 치명적 에러 발생 위험.
  * **최종 합의:** **승인 및 방어 로직 도입**
  * **해결책 상세 (Sanity Check):** 파이썬 스크립트 실행 시 추출 작업 전 가장 먼저 `PRAGMA table_info(ZBOOKNOTE)` 명령을 실행. 우리가 사용하는 핵심 컬럼(`ZNOTEID`, `ZCOMMENTTEXT` 등)의 존재 여부를 검사하는 무결성 검증 단계를 거침. 스키마 변경이 감지되면 즉각 동기화를 중단하고 Slack/Telegram 봇으로 **"🚨 [긴급] MarginNote DB 스키마 변경 감지! 데이터 꼬임을 막기 위해 동기화를 강제 중단합니다"** 라는 강력한 알람 송출.

* **9.1.3 로컬 PC(Mac) 서버 의존성**
  * **상세 내용:** 폰에서 내린 명령을 처리하기 위해 Mac이 상시 서버처럼 작동해야 함. 외출 시 절전모드 등으로 인한 가용성 공백 우려.
  * **최종 합의:** **문제 없음 (클리어)**
  * **사유:** 사용자는 헤르메스 에이전트 가동을 위해 Mac을 상시 켜두는 환경을 유지하므로, 시스템 휴지기(Downtime) 이슈는 발생하지 않음.

* **9.1.4 AnkiConnect 포트 침묵 실패(Silent Fail)**
  * **상세 내용:** Mac에서 자동화 스크립트는 도는데 정작 Anki 프로그램이 꺼져있거나 크래시 나서 포트(8765)가 닫힌 경우, 에러 없이 데이터 동기화가 누락되는(Silent Fail) 치명적 문제.
  * **최종 합의:** **승인 및 자동 복구/알람 도입**
  * **해결책 상세 (Health Check):** 파이썬 동기화 스크립트 구동 시 `http://localhost:8765` 로 핑(Ping) 테스트 수행. 무응답 시 Mac OS 내부 명령어(`open -a Anki`)를 호출해 백그라운드에서 Anki를 강제 실행. 만약 재실행 후에도 핑 실패 시, Slack 봇을 통해 **"🚨 Anki 앱이 꺼져 있어 동기화가 중단되었습니다."** 라고 즉시 알람을 발생시켜 침묵 실패를 완벽히 차단.

### 9.2 데이터 정합성 및 백업 시스템 리스크

* **9.2.1 단방향 동기화 및 모바일 수정 데이터 증발**
  * **상세 내용:** 스마트폰(AnkiDroid)에서 데이터를 수정한 후 PC와 동기화 시, 충돌 병합 없이 PC 원본 데이터로 덮어씌워져 모바일 수정 내역이 날아감.
  * **최종 합의:** **수용 (클리어)**
  * **사유:** "안드로이드(폰)에서는 오직 읽고 복습만 하며 절대 카드 내용을 수정하지 않는다"는 강력한 사용자 행동 원칙으로 해결.

* **9.2.2 분산 DB 간 시점 불일치(Time-Sync Mismatch)**
  * **상세 내용:** MarginNote DB, Obsidian 폴더, Anki DB 3개가 각각 구동됨. 시스템 장애 시 5TB 구글 드라이브에서 백업본을 롤백할 때, 특정 앱만 예전 백업본으로 덮어씌우면 DB 간 참조가 깨져 고아 데이터(Orphan Data)가 대량 발생.
  * **최종 합의:** **승인 및 원자적 백업(Atomic Backup)**
  * **해결책 상세:** 백업을 수행할 때 스크립트가 1) 마진노트 `.marginnotes` 원본, 2) Obsidian Vault 마크다운 폴더, 3) Anki `collection.anki2` 파일을 **반드시 한 날 한 시에 하나의 단일 `.zip` 파일**로 압축하여 저장함. 문제 발생 시 사용자는 해당 통합 zip 파일 압축을 풀어 3곳에 동시에 덮어씌움으로써 시점 불일치를 원천 차단.

* **9.2.3 개정판 출시에 따른 연결망 붕괴 (Stale Data)**
  * **상세 내용:** 오답 연결 시 목차 번호(예: `1.1.1 자산손상`)에 과도하게 의존할 경우, 내년 교재 개정으로 번호가 밀리면 기존에 쌓인 Obsidian 약점 랭킹 통계와 연결망이 모두 붕괴됨.
  * **최종 합의:** **승인 및 버전 관리 파일 분리**
  * **해결책 상세:** 시스템 내부나 에이전트 프롬프트에 `1.1.1`을 하드코딩하지 않음. 대신 `TOC_Mapping.json` 이라는 별도의 맵핑 파일을 생성. 
  에이전트는 사용자 명령 시 무조건 이 json 파일을 조회하여 목차를 해석함. 교재 개정 시 json 파일 내 번호표만 한 번 싹 고쳐주면, 기존 연결망을 훼손하지 않고 전체 시스템이 새 교재 버전에 즉각 동기화됨.

* **9.2.4 수동 조작 시 UUID 파괴 및 카드 중복 증식**
  * **상세 내용:** 파이썬 스크립트가 카드의 중복 방지를 위해 심어둔 마진노트 고유 ID(`ZNOTEID`)를, 사용자가 Anki 내에서 수동으로 '카드 복사' 또는 '잘라내기(Split)' 기능으로 건드릴 경우 UUID가 파괴되거나 복제되어 카드가 무한 증식하는 문제.
  * **최종 합의:** **수용 (클리어)**
  * **사유:** "Anki Desktop 내에서는 절대 시스템 생성 카드를 수동으로 쪼개거나 내용을 복사 생성하지 않는다"는 대원칙 하에 자연스럽게 방어. 분할이 필요하면 오직 MarginNote 원본에서 분할함.

### 9.3 AI 에이전트 및 RAG 파이프라인 리스크

* **9.3.1 전문 용어 STT 변환의 품질 저하 (Data Poisoning)**
  * **상세 내용:** 회계/세법 강의의 복잡한 한자어 및 강사의 억양으로 인해 STT가 오탈자를 만들고, 이 오염된 텍스트가 Tutor 에이전트의 지식 베이스(ChromaDB)에 적재되어 심각한 환각(Hallucination)을 유발할 위험.
  * **최종 합의:** **승인 및 Ground-Truth 기반 오토 코렉트 파이프라인 구축**
  * **해결책 상세:** 이미 보유 중인 가장 완벽한 텍스트 원본(교재 PDF 텍스트 및 마진노트 데이터)에서 **'사용자 전용 용어 사전(Custom Dictionary)'**을 자동 추출함. STT 텍스트가 RAG DB에 적재되기 전 중간 필터 역할을 하는 소형 LLM이 이 사전을 기반으로 오탈자("자본유지 젓갈법" -> "자본유지 접근법")를 강제 치환 및 정제한 후 최종 저장함.

* **9.3.2 요약 에이전트로 인한 미세 컨텍스트 소실**
  * **상세 내용:** Memory Editor 에이전트가 1만 토큰의 긴 슬랙 대화를 단축시키기 위해 핵심만 뽑아 한 줄로 요약하는 과정에서, 세법의 미세한 예외 조건이나 사용자 오답의 섬세한 뉘앙스가 영구 소실됨.
  * **최종 합의:** **승인 및 구조화된 프롬프트 양식(JSON/YAML) 강제**
  * **해결책 상세:** Memory Editor 에이전트에게 절대 자유도 높은 산문형 한 줄 요약을 허용하지 않음. 하드디스크(`LLM_Wiki/User_State.md`)에 기록할 때 아래와 같은 구조화된 포맷 출력을 강제함.
    * `[약점 개념]`: 단기매매금융자산
    * `[결정적 조건/가정]`: 시장성 상실 시
    * `[사용자의 실수 원인]`: 평가손익을 당기손익이 아닌 OCI로 잘못 인식함

* **9.3.3 Checker/Tutor 교차 검증의 무한 루프**
  * **최종 합의:** **보류 (패스)**
  * **사유:** Claude와 Qwen이 서로 합의하지 못하고 교착 상태에 빠지는 것을 막기 위한 서킷 브레이커 로직은 현 단계 구현이 다소 복잡하므로 사용자 피로도를 고려하여 우선순위에서 배제함.

* **9.3.4 정규표현식 이탈에 따른 API 토큰 과금 누수**
  * **최종 합의:** **보류 (패스)**
  * **사유:** 오타 발생 시 Levenshtein 거리를 계산하여 무료 자동 교정하는 방안은 현재로선 비용적 타격이 크지 않아 우선순위에서 배제함.

### 9.4 카드 생성 규칙 및 스케줄링 간섭 리스크

* **9.4.1 무분별한 카드 증식 방어 (Card Bloat)**
  * **상세 내용:** 문서의 평범한 문장(`~은 ~이다`)을 빈칸 카드로 만들거나, 4단계 절차 트리거 시 카드가 4장으로 쪼개져 Anki 리뷰 지옥을 초래.
  * **최종 합의:** **승인 및 절차 카드 단일화(Progressive Reveal)**
  * **해결책 상세:** 
    1. **정의 자동 감지 폐지:** 코드가 임의로 카드를 만들지 않게 삭제.
    2. **점진적 공개 카드:** 4단계 `절차:` 트리거 사용 시 안키 카드를 단 1장만 생성함. 단, 안키 카드 뒷면의 Javascript 템플릿을 수정하여, 사용자가 화면을 터치할 때마다 1단계, 2단계 텍스트가 순차적으로 부드럽게 나타나게 구현함. 복습 횟수는 1번으로 유지하며 순서 암기 효과는 극대화.

* **9.4.2 스케줄러 에이전트의 FSRS 알고리즘 훼손**
  * **상세 내용:** 에이전트가 SQLite 상의 `due`(복습 예정일)를 강제로 밀어버리면 FSRS 망각 곡선 알고리즘이 붕괴되어 다음 복습 주기를 기하급수적으로 뻥튀기함.
  * **최종 합의:** **승인 및 Postpone API 전용 연동**
  * **해결책 상세:** 에이전트는 절대 DB 원시 데이터를 직접 조작할 수 없도록 권한을 박탈당함. 대신 Anki 공식 생태계의 **'FSRS Helper 연기(Postpone) 기능' API**만 호출하도록 허용됨. 

* **9.4.3 이미지 가림(Image Occlusion)의 기술적 한계**
  * **상세 내용:** MarginNote 코멘트에 `가림:` 이라고만 치면 파이썬 스크립트가 이미지 내의 어떤 좌표(x, y)를 박스로 가려야 할지 모름.
  * **최종 합의:** **해결 및 당일 수동 워크플로우 정립**
  * **해결책 상세:** 컴퓨터 비전 강제 추적 오버엔지니어링을 폐기. 파이썬은 그저 마진노트 이미지를 넣은 **가림 전용 뼈대 카드**만 Anki로 넘김. 사용자는 강의 수강 **당일**, Anki Desktop을 켜서 마우스로 직접 빨간 박스를 치는 수동 작업을 정규 스케줄로 편입시킴.

### 9.5 시스템 UX 및 인지적 과부하 리스크

* **9.5.1 탭(Tab) 들여쓰기에 대한 과도한 파싱 의존성**
  * **상세 내용:** Logseq에서 사용자가 탭이나 스페이스바를 한 번 실수하면 2차 연습서 문제의 부모(공통자료)-자식(물음) 구조가 와해됨.
  * **최종 합의:** **해결 완료 (사용자 탭핑 불필요)**
  * **해결책 상세:** 사용자가 MarginNote 앱 내에서 문제집을 마인드맵으로 묶어두면 DB(`ZGROUPNOTEID`)에 완벽한 위계 구조가 저장됨. 파이썬 스크립트는 이 마진노트의 본원적 계층 구조를 최우선 진실로 삼아 읽어와, 컴퓨터가 알아서 들여쓰기 된 Logseq 마크다운 파일을 자동 생성해주므로 사용자는 탭을 칠 필요가 전혀 없음.

* **9.5.2 목차 암기 및 호명의 인지적 스트레스**
  * **상세 내용:** 틀린 개념을 헤르메스와 연결할 때 기본서 목차 번호(예: 1.1.1)를 암기하고 타이핑해야 하는 스트레스.
  * **최종 합의:** **승인 및 자동 추천/승인 UX 도입**
  * **해결책 상세:** 목차 번호 암기 폐지. 에이전트는 연습서 문제의 원본 텍스트("상각원가" 등)를 추출해 교재 목차 DB와 벡터 검색(유사도 매칭)을 수행함. 슬랙 봇에 **"🤖 힌트: 이 문제는 [6.3.2 당기손익-공정가치측정] 개념과 가장 유사합니다. 연결할까요?"** 와 함께 상위 3개의 후보 버튼을 띄움. 후보군에 없다면 **`[🔍 직접 찾기]`** 버튼으로 대체.

* **9.5.3 지나치게 복잡한 트리거 규칙 제약 (주객전도 현상)**
  * **상세 내용:** 회계학 공부보다 9가지 트리거 규칙(`주의:`, `강사:`)을 외우고 코멘트에 타이핑하는 데 귀중한 에너지를 소모함.
  * **최종 합의:** **승인 및 색상 기반 초격차 단순화 달성**
  * **해결책 상세:** 마진노트 SQLite DB(`ZBOOKNOTE` 테이블) 내에 색상 정보 컬럼(`ZCOLOR`)이 인덱스 숫자로 명확히 저장됨. 수동 텍스트 타이핑을 폐지하고 가장 직관적인 UX를 도입. 파이썬 스크립트에 **"빨간색(Index 2) 노드 = 무조건 `⚠주의` 카드 생성", "초록색(Index 3) 노드 = 무조건 `📋절차` 카드 생성"** 코드를 추가. 사용자는 터치/펜슬로 노드 색깔만 칠해두면 시스템이 알아서 카드의 종류와 트리거를 분류.
