# CPA 학습 자동화 시스템 — 궁극의 통합 인수인계 마스터플랜 (Ultimate Grand Masterplan)

> **💡 [문서 개요]**
> 본 문서는 기존에 작성되었던 시스템 마스터플랜과, 시스템 운영 중 도출된 **모든 구조적/UX적 한계점 및 최종 합의된 방어책(Mitigation)**을 단 하나의 내용 유실도 없이 완벽하게 병합한 **최종의 최종본(Single Source of Truth)**입니다.
> 처음 시스템을 접하는 담당자나 개발자가 코어 아키텍처부터 구체적인 DB 테이블, 안키 스케줄링 로직, 에이전트 연동, 그리고 **과거의 시행착오를 극복한 초격차 UX 단순화 설계**까지 한눈에 파악하고 즉각 구현할 수 있도록 가장 상세하게 작성되었습니다.

---

## 1. 프로젝트 개요 및 4대 코어 아키텍처 

CPA 학습 시스템은 수작업 암기를 넘어, **"내가 무엇을 알고 모르는지(메타인지)"를 완벽하게 통제**하고, 학습 효율을 극대화하기 위한 자율형 AI 기반의 사이보그 컨설턴트 환경입니다. 데이터는 '구조 뼈대 생성'부터 '약점 역추적'까지 완벽한 순환 고리(Circular Workflow)를 형성합니다.

### 1.1 시스템 전체 구조도 및 파이프라인

```mermaid
graph TD
    A[기본서 TOC 및 문항 자르기 시스템] -->|아웃라이너 뼈대 OPML 생성| B[MarginNote 4 역삽입 Import]
    B -->|사용자 학습: 시각적 트리 구성 및 색상 지정| C[MarginNote SQLite DB]
    C -->|Python 1차 변환| D[Logseq & Obsidian 마크다운 Vault]
    D -->|Python 2차 변환| E[Anki 자동 복습기]
    E -->|오답(Lapse) 통계 발생| F[Obsidian 대시보드 랭킹 산출]
    F -->|에이전트 추천 연동| E
```

### 1.2 시스템별 상세 역할 분담

| 시스템 | 역할 (기능) | 상세 설명 |
|--------|-----------|---------|
| **MarginNote4** | **창고 & 입력단** | 교재, 필기, 오답이 모이는 원천 DB. 앱의 내부 트리 구조(`ZGROUPNOTEID`)와 색상(`ZCOLOR`)만으로 모든 로직을 제어합니다. |
| **Logseq/Obsidian** | **계층 통제 및 관제실** | 추출된 데이터를 기반으로 실시간 약점 랭킹(대시보드)을 제공하며, LLM Wiki의 영구 저장소 역할을 합니다. |
| **Anki** | **자동 복습기** | FSRS 망각곡선이 100% 스케줄을 통제합니다. 모바일(Android)에서는 **수정 불가(읽기/복습 전용)** 정책을 강제합니다. |
| **AI (Hermes 등)** | **LLM 지식 베이스** | 공유 폴더의 Markdown을 읽고 목차 자동 매핑, 환각 팩트체크 등 다중 에이전트 역할을 수행합니다. |

## 2. 도구 설치 현황 및 시스템 가용성 리스크 방어

### 2.1 설치된 도구 (작성일: 2026-06-06 기준)
* Anki 25.09 (`/Applications/Anki.app`)
* Obsidian 1.12.7 (`/Applications/Obsidian.app`)
* MarginNote 4 (`/Applications/MarginNote 4.app`)
* AnkiConnect 플러그인 (포트: 8765)

### 2.2 🚨 [인프라 가용성 리스크 및 방어책]
* **Mac 상시 가동:** 백그라운드 에이전트(Hermes)를 위해 Mac은 상시 켜두는 것을 원칙으로 합니다.
* **MarginNote URL 종속성:** 모든 링크는 `marginnote4app://` 전용 스킴을 사용합니다. 애플 생태계 종속성은 어쩔 수 없는 한계로 수용합니다.
* **✨ [방어책] AnkiConnect 침묵 실패(Silent Fail) 차단:**
  * Mac 스크립트 구동 시, Anki가 꺼져있으면 동기화가 에러 없이 누락되는 문제를 막기 위해 **Health Check 로직**을 강제합니다.
  * 파이썬 스크립트는 동기화 전 `http://localhost:8765` 로 Ping을 전송하고, 응답이 없으면 OS 명령어(`open -a Anki`)로 Anki를 강제 백그라운드 실행합니다. 재시도 실패 시 Slack으로 긴급 알림을 보냅니다.

## 3. 마진노트(MarginNote4) 데이터베이스 아키텍처 상세

스크립트는 마진노트의 SQLite `.marginnotes` 테이블을 직접 파싱합니다. 

### 3.1 🚨 [방어책: DB 스키마 무결성 검증 (Sanity Check)]
비공식 DB 파싱은 앱 업데이트 시 구조가 바뀌면 치명적입니다. 이를 막기 위해 파이썬 스크립트 실행 최상단에 **무결성 검사 코드**를 추가합니다.
* `PRAGMA table_info(ZBOOKNOTE)`를 실행하여 `ZNOTEID`, `ZCOLOR`, `ZCOMMENTTEXT`, `ZGROUPNOTEID` 컬럼의 존재를 확인합니다.
* 스키마 변경 감지 시 즉각 동기화를 중단하고 봇으로 🚨"DB 스키마 변경 감지! 데이터 꼬임 방지를 위해 중단됨" 알림을 송출합니다.

### 3.2 핵심 테이블 명세
| 테이블 / 컬럼 | 내용 및 활용 용도 |
|--------------|----------------|
| `ZBOOKNOTE.ZNOTEID` | 노드 고유 ID (UUID). Anki 카드 중복 방지에 사용됨. |
| `ZBOOKNOTE.ZGROUPNOTEID` | **[✨ 핵심]** 부모 노드 ID. 이를 통해 파이썬이 Logseq의 들여쓰기 탭(Tab)을 사용자 대신 완벽하게 자동 생성합니다. |
| `ZBOOKNOTE.ZCOLOR` | **[✨ 초격차 UX]** 노드의 색상 인덱스(예: 빨강=2, 초록=3). 귀찮은 텍스트 트리거 대신 색상만으로 Anki 카드 종류를 결정합니다. |
| `ZBOOKCOMMENT.ZCOMMENTTEXT` | 특수 입력이 필요할 때 사용하는 코멘트 내용. |

## 4. Phase별 초정밀 시스템 실행 계획 및 고도화 (Execution Plan)

### [Phase 0] 교재 해체 및 TOC 파이프라인 (버전 관리 도입)
* **목차 번호 하드코딩 탈피:** 내년 교재 개정 시 연결망 붕괴(Stale Data)를 막기 위해 `TOC_Mapping.json` 파일을 도입합니다. 에이전트는 무조건 이 JSON을 거쳐 목차를 찾으므로, 개정 시 숫자만 한 번 바꿔주면 시스템 전체가 새 버전에 동기화됩니다.

### [Phase 1~2] 마크다운 변환 및 들여쓰기 스트레스 완벽 제거
* **✨ [개선] 탭(Tab) 들여쓰기 파싱의 사용자 개입 제로화:**
  과거에는 사용자가 직접 Logseq에서 탭을 쳐서 위계를 잡으려 했으나 오류율이 높았습니다. 이제는 사용자가 마진노트 마인드맵 상에 그려둔 본원적 계층 구조(`ZGROUPNOTEID`)를 시스템이 100% 신뢰(Source of Truth)하여 **컴퓨터가 알아서 마크다운 탭 들여쓰기를 자동 생성**합니다.

### [Phase 3~4] Anki 동기화 및 강력한 RAG 파이프라인
* **✨ [개선] 전문 용어 STT 오탈자 방어 (Data Poisoning 차단):**
  강의 녹음(STT)을 그대로 DB에 넣지 않습니다. 교재와 마진노트에서 추출한 **'사용자 전용 용어 사전(Custom Dictionary)'**을 필터로 둡니다. STT 텍스트가 RAG DB에 적재되기 전 소형 LLM이 이 사전을 기반으로 오탈자("자본유지 젓갈법" -> "자본유지 접근법")를 강제 치환 및 정제합니다.

### [Phase 5] 카파시(Karpathy) LLM OS와 헤르메스 에이전트 확장
* **✨ [개선] 요약 에이전트의 미세 뉘앙스 보존 (JSON 강제):**
  긴 대화를 요약하는 Memory Editor가 회계학의 세밀한 가정을 날려먹는 것을 방지합니다. 1줄 요약을 금지하고 아래의 YAML/JSON 구조화 출력을 강제합니다.
  * `[약점 개념]`: 단기매매금융자산
  * `[결정적 조건/가정]`: 시장성 상실 시
  * `[사용자의 실수 원인]`: OCI가 아닌 당기손익으로 오분류함.
* **✨ [개선] 스케줄러 에이전트의 FSRS 망각곡선 훼손 방어:**
  일정 변경 시 에이전트가 Anki DB의 날짜(`due`)를 임의 조작하는 것을 금지합니다. 오직 수학적 페널티를 보정 분산시켜 주는 공식 **'FSRS Helper 연기(Postpone) API'**만 호출하도록 권한을 극도로 제한합니다.

### [Phase 6] 5TB 무제한 스냅샷 및 시점 불일치 방어
* **✨ [개선] 원자적 백업 (Atomic Backup):**
  복구 시 고아 데이터(Orphan Data) 꼬임을 막기 위해, 백업 스크립트는 1) MarginNote DB, 2) Obsidian Vault, 3) Anki `collection.anki2`를 **반드시 한 날 한 시에 단일 `.zip` 파일**로 압축하여 구글 드라이브에 올립니다.

## 5. 카드 생성 규칙, 데이터 허용 형태 및 초격차 UX 트리거

### 5.1 🚨 [과거의 피로도: 9대 텍스트 트리거 폐기 및 단순화]
과거 수험생이 `주의:`, `강사:` 등을 매번 치면서 공부보다 시스템 규칙 암기에 에너지를 쏟는 주객전도 현상이 있었습니다. 
이를 해결하기 위해 **마진노트 DB의 `ZCOLOR` 인덱스**를 활용한 궁극의 단순화를 달성했습니다.

**[새로운 색상 기반 자동 트리거 매핑]**
사용자는 글씨를 칠 필요 없이 펜슬/마우스로 마진노트 노드 색깔만 칠하면 됩니다!
* **빨간색 노드 (Index 2):** 자동으로 ⚠ **[함정/주의 카드]** 생성 (짧은 스케줄 적용)
* **파란색 노드 (Index 4):** 자동으로 🎤 **[강사 팁 카드]** 생성
* **초록색 노드 (Index 3):** 자동으로 📋 **[절차 카드]** 생성

### 5.2 특수 카드 로직의 고도화 방안
1. **무분별한 카드 증식(Card Bloat) 방어:**
   * 평범한 문장을 빈칸으로 만들던 '정의 자동 감지' 코드를 폐기합니다.
2. **절차 카드의 점진적 공개 (Progressive Reveal):**
   * 4단계 절차를 4장의 Anki 카드로 쪼개어 리뷰 지옥을 만들지 않습니다. **단 1장의 카드**만 생성하되, Anki 뒷면 Javascript 템플릿을 수정하여 화면을 터치할 때마다 1단계, 2단계 텍스트가 스르륵 순차적으로 나타나도록 구현합니다.
3. **가림 (Image Occlusion) 카드의 완벽한 수동 워크플로우 정립:**
   * 기술적 한계: 파이썬은 컴퓨터 비전이 없어 이미지 안의 어느 X,Y 좌표를 빨갛게 가려야 할지 모릅니다.
   * **최종 워크플로우:** 파이썬 스크립트는 원본 이미지를 담은 '가림 뼈대 카드'만 Anki로 넘깁니다. 사용자는 강의 수강 당일, Anki Desktop을 켜서 마우스로 직접 빨간 박스를 치는 수동 작업을 진행합니다. 이 수동 작업이 끝나면 카드가 정상적으로 FSRS 스케줄에 진입합니다.

## 6. 메타인지 추적 및 약점 연결망 구축 (인지적 과부하 제로)

2차 연습서 문제의 근본 원인을 연결하는 과정에서 "목차 번호(예: 1.1.1)를 외워야 하는 스트레스"를 기술로 완벽히 제거했습니다.

### 6.1 에이전트 기반 자연어 검색 및 추천 UX 도입
사용자는 에이전트에게 숫자를 말할 필요가 없습니다.
> 🗣️ 사용자: "헤르메스, 방금 푼 연습서 문제랑 그 원인이 되는 핵심 개념 연결해 줘."

1. **자동 추천 (Vector Search):** 에이전트가 연습서 문제에 포함된 핵심 키워드("상각원가", "손상" 등)를 바탕으로 `TOC_Mapping.json`을 벡터 검색합니다.
2. **원클릭 승인 UI:** 슬랙 봇에 알림이 옵니다. *"🤖 이 문제는 [6.3.2 당기손익-공정가치측정] 개념과 가장 유사합니다. 연결할까요? [✅ 승인]"*
3. **직접 검색 풀백 (Fallback):** 만약 정답 후보가 없다면, `[🔍 직접 찾기]` 버튼을 눌러 에이전트에게 "아니, 자본유지 접근법 찾아줘"라고 텍스트로 치면 해당 목차를 즉시 찾아 연결합니다.

## 7. 시스템 방어망 및 관리자 상세 (Defense & Mitigations)

통합 시스템이 크래시 없이 안정적으로 영구 운영되기 위한 최후의 3대 안전 기제입니다.

1. **모바일 수정 금지 원칙 (단방향 덮어쓰기 방어):** 
   스마트폰(Android AnkiDroid)에서는 데이터 수정 시 덮어쓰기로 날아갑니다. 따라서 폰에서는 오직 '읽기 및 복습'만 진행하고, 카드 수정은 Slack 봇(Inbox)이나 귀가 후 Mac에서 처리합니다.
2. **Anki 앱 내 수동 분할(Split) 영구 금지 (UUID 파괴 방지):**
   시스템은 중복 카드 방지를 위해 MarginNote의 `ZNOTEID`를 Anki 뒷면에 숨겨둡니다. Anki Desktop에서 사용자가 수동으로 카드를 복사하거나 자르면 이 ID가 파괴되어 무한 증식 에러가 발생합니다. **모든 카드 분할과 구조 변경은 오직 MarginNote 원본에서 수행해야 합니다.**
3. **중복 감지 방어 코드:** 
   혹시라도 수동 복제된 카드가 생기면, 파이썬이 동기화 시 동일한 `ZNOTEID`를 2장 이상 발견할 경우 즉각 동기화를 멈추고 에러를 띄우도록 로직을 세팅합니다.

## 8. 부록 (Appendix): 설계 철학 및 폐기된 과거의 아이디어

향후 시스템 확장 담당자는 아래 사용자님의 철학에 반하는 과도한 개발(Over-engineering)을 시도하지 않아야 합니다.

* **에이전트 간 무한 교착(Deadlock) 제어 보류:** Claude와 Qwen이 서로 반박하며 핑퐁을 칠 때 강제로 끊는 로직은, 현재 피로도와 비용을 고려해 굳이 구현하지 않고 패스(수용)하기로 합의함.
* **정규식 오타의 LLM 파싱 (비용 누수):** 오타 교정을 위해 Fuzzy Matching(편집 거리 알고리즘)을 도입하는 방안도, 에이전트 비용이 크지 않아 우선순위에서 배제함.
